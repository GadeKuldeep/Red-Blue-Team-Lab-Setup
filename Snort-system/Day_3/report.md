# SSH Attack Detection and Custom Snort Rule Development

## Overview

This lab focused on creating an environment for SSH attack simulation and analyzing the generated logs using Snort IDS.

The objective was to understand how SSH-related activity appears in logs, observe Snort detections, and create custom rules based on the observed traffic to improve detection capabilities and SOC analyst skills.

# Firewall Configuration

Configured default firewall policies on Ubuntu Server.

Block incoming traffic by default:

```bash
sudo ufw default deny incoming
```

Allow outgoing traffic:

```bash
sudo ufw default allow outgoing
```

---

# SSH Configuration

Allowed SSH service through the firewall.

```bash
sudo ufw allow 22/tcp
```

Purpose:

* Open SSH port
* Allow SSH communication for testing

---

# Enable SSH Service

Enabled and started SSH service on Ubuntu Server.

```bash
sudo systemctl enable --now ssh
```

Purpose:

* Allow SSH connections from Kali Linux
* Generate logs for monitoring and analysis

---

# Attack Simulation

Performed SSH connection attempts from Kali Linux to Ubuntu Server.

Purpose:

* Generate SSH traffic
* Observe Snort detections
* Analyze network logs

---

# Snort Log Analysis

Observed alerts generated by Snort from previously configured detection rules.

Existing rule used for TCP scan detection:

```snort
alert tcp any any -> 192.168.1.16 any (
msg:"TCP Scan Detected";
flags:S;
threshold:type threshold, track by_src, count 20, seconds 3;
sid:10000003;
rev:1;
)
```

Generated alert:

```text
TCP Scan Detected
```

This rule helped identify suspicious TCP scanning activity directed toward the target machine.

---

# Custom SSH Detection Rule

After reviewing the generated traffic and logs, created a custom rule specifically for SSH monitoring.

Rule:

```snort
alert tcp any any -> 192.168.1.16 22 (
msg:"SSH Port Under Attack";
sid:10000008;
rev:1;
)
```

Purpose:

* Monitor traffic directed toward SSH port 22
* Generate alerts for SSH-related activity
* Improve visibility into SSH attacks

---

# Alert Verification

Reloaded Snort configuration and generated SSH traffic again.

Observed successful alert generation from the newly created rule.

Generated alert:

```text
SSH Port Under Attack
```

This confirmed that the rule was functioning correctly and detecting SSH traffic as intended.
