# Red-Blue-Team-Lab-Setup
# CyberOps SOC Home Lab – Red Team / Blue Team Project

## Overview

This project demonstrates a **Red Team vs Blue Team cybersecurity home lab** designed to simulate real-world attack scenarios and SOC monitoring workflows.

The lab environment allows an attacker machine to launch various reconnaissance and exploitation techniques against a vulnerable target while a Security Operations Center (SOC) monitors and analyzes the generated security events.

The primary objective of this project is to practice **attack simulation, SIEM monitoring, incident detection, and security analysis** in a controlled virtual environment.

---

## Lab Architecture


**View the full interactive architecture diagram:**

👉 https://gadekuldeep.github.io/Red-Blue-Team-Lab-Setup/


The environment consists of two virtual machines connected through a **VirtualBox bridged network**, enabling realistic network communication and traffic simulation.

**Attacker Node**

* Kali Linux
* IP: `192.168.1.16`
* Used for offensive security testing and attack simulation.

**Defender / SOC Node**

* Ubuntu Server
* Hosts monitoring and vulnerable services for detection analysis.

**Network**

* Bridged Network: `192.168.1.0/24`

Architecture diagram:

```
Kali Linux (Attacker)
192.168.1.16
        │
        │  Attack Traffic
        ▼
Ubuntu Server (Defender / SOC)
Wazuh SIEM + DVWA
192.168.1.X
```

## Detection and Monitoring

Security monitoring is performed using **Wazuh SIEM**, which collects logs and generates alerts based on suspicious activity.

Detection workflow:

1. Attack traffic originates from the Kali Linux machine.
2. The Ubuntu server receives the attack attempts.
3. Logs are generated by system services and applications.
4. Wazuh analyzes logs and triggers alerts.
5. The SOC analyst reviews alerts in the Wazuh dashboard.

## Attack Scenarios Simulated

The lab is used to simulate common attack techniques including:

* Network reconnaissance
* Port scanning
* Web vulnerability scanning
* SQL injection attempts
* SSH brute force attacks
* Web exploitation against DVWA

These attacks generate logs and alerts that can be analyzed in the SIEM dashboard.

## Incident Reporting Methodology

Each attack simulation is documented as a **daily incident report**.

For every scenario, the following information is recorded:

* Attack description
* Tool used
* Target system
* Evidence screenshots
* Wazuh alerts
* SOC analyst observations
* Mitigation recommendations

Project structure example:

```
daily-incidents/
   day-01-reconnaissance/
   day-02-bruteforce/
   day-03-sql-injection/
```

Each folder contains:

* Attack screenshots
* Wazuh alert screenshots
* Incident analysis report

## Example SOC Incident Workflow

1. Attacker performs reconnaissance using Nmap.
2. Target system logs network activity.
3. Wazuh SIEM analyzes the logs.
4. Alerts appear on the Wazuh dashboard.
5. SOC analyst investigates the alerts.
6. Incident report is written.
